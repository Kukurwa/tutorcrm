# TutorCRM — Roadmap

Стратегия: **сначала весь фронтенд на моках, затем backend замещает моки**. Моки живут внутри Next.js Route Handlers (in-memory + фикстуры) или MSW — фронт обращается к своему API как к настоящему, только реализация внутри — фейковая. `packages/contracts` (zod) фиксируется в самом начале и служит контрактом, который одинаково соблюдают и моки, и реальные реализации.

Этапы идут последовательно. Каждый чек-пункт — единица работы, которая пишется в `docks/CHANGELOG.md` после выполнения. Не запускать работу по задачам следующего этапа, пока не закрыт текущий.

---

# БЛОК 1 — FRONTEND НА МОКАХ

На этом блоке нет PostgreSQL, Prisma, BullMQ, реальных мессенджеров и реального Socket.IO-процесса. Auth — NextAuth с JWT strategy (без БД), «БД» — in-memory фикстуры. Realtime — клиентская симуляция (таймеры, event-bus в памяти). Цель блока — сдача визуальной версии продукта с кликабельными сценариями.

# БЛОК 2 — BACKEND (замещение моков)

Контракты (zod-схемы из `packages/contracts`) уже есть. Задача блока — заменить in-memory моки на реальные реализации: БД, workers, realtime, интеграции. Фронт не меняется: те же Route Handler-ы, та же сигнатура.

---

- [ ] `apps/worker`: Node.js + BullMQ consumer, caverage-подключение к `packages/core`, регистратор cron.
- [ ] `apps/realtime`: Node.js + Socket.IO, авторизация по NextAuth session cookie, Redis pub/sub.
- [ ] Фронт: замена `lib/fake-realtime.ts` на реальный Socket.IO-клиент.
- [ ] Rate-limiting на auth-ручках.

**Приёмка**: логин через реальный NextAuth + Prisma; тестовое событие от `worker` долетает клиенту через `realtime`.

# TutorCRM — ТЗ по фронтенду v1.0

Источник требований: `BUSINESS_LOGIC_Client.pdf` (бизнес-логика v1.0).
Модуль интеграции с Telegram (UI-часть подключения аккаунтов, работы Userbot/Bot) — будет уточнён отдельным документом.

---

## 1. Общие положения

### 1.1 Назначение
Веб-приложение CRM для посредника между клиентами и репетиторами. Три роли пользователей: Admin, Dispatcher, LeadGen. Основная рабочая единица — диспетчер, который ведёт переписку и воронку заявок.

### 1.2 Стек
- **Фреймворк**: **Next.js 14+ (App Router)** + TypeScript. Страницы — RSC по умолчанию, Client Components — для интерактивных разделов (Inbox, Kanban, формы).
- **Маршрутизация**: App Router (`app/` directory) с группировкой `(auth)` / `(app)` для раздельных layout-ов.
- **Auth**: **NextAuth.js** (credentials provider), сессии в БД через Prisma adapter.
- **State / data**: **TanStack Query** (клиентские кэши серверных данных) + **Zustand** для UI-state. Серверные компоненты читают данные напрямую через сервисы `packages/core`.
- **Realtime**: **Socket.IO** client (WS-сервер — отдельный процесс `apps/realtime`, см. `TZ_BACKEND.md`).
- **UI-kit**: **shadcn/ui** (Radix) + **Tailwind CSS**. Единственный UI-кит, не смешивать с другими.
- **Формы**: **react-hook-form + zod**. zod-схемы шарятся с API через `packages/contracts`.
- **Таблицы**: TanStack Table.
- **Календарь**: FullCalendar или react-big-calendar.
- **Kanban**: dnd-kit.
- **Графики**: Recharts или ECharts.
- **i18n**: `next-intl` (минимум RU; предусмотреть подключение UA/EN).
- **Иконки**: lucide-react.
- **Storybook**: обязателен для `packages/ui`.
- **Тесты**: Vitest + React Testing Library; Playwright для e2e ключевых сценариев.

### 1.3 Платформа и браузеры
- Desktop-first (основной рабочий инструмент диспетчера).
- Адаптив: корректная работа от 1280×800; для ноутбуков 1366×768 — без горизонтального скролла.
- Поддержка последних 2 версий Chrome, Edge, Firefox, Safari.
- Мобильная вёрстка — не в MVP (кроме страницы логина и минимального списочного режима Inbox — по согласованию).

### 1.4 Технические требования
- Тёмная/светлая темы (переключатель в настройках пользователя, `next-themes`).
- Сессии — NextAuth в HttpOnly cookie; SSR-ready (middleware проверяет сессию до рендера страниц).
- Все клиентские обращения к API — через единый типизированный клиент поверх Route Handlers (`lib/api-client.ts`) с перехватом 401/403/419 и централизованной обработкой ошибок.
- Серверные компоненты — читают данные через сервисы из `packages/core`, не делая HTTP-запросов на свои же ручки.
- Глобальный `error.tsx` + `not-found.tsx` + `global-error.tsx` (Next.js App Router).
- Code-splitting — автоматический по роутам; динамические импорты для тяжёлых виджетов (Kanban, календарь, графики).
- Оффлайн-состояние: баннер «нет связи», очередь отправки сообщений — по согласованию (в MVP: просто блокировка с индикатором).
- Middleware (`middleware.ts`): проверка сессии на защищённых маршрутах, редирект на `/login`.

## 2. Роли и доступ

| Роль        | Основные возможности                                                                                                   |
|-------------|------------------------------------------------------------------------------------------------------------------------|
| Admin       | Всё, что делают другие роли + управление пользователями, настройками, тарифами комиссий, шаблонами, справочниками, SLA |
| Dispatcher  | Переписка, лиды, заявки, публикации, назначения, контракты, финансовые операции частично, «своё» на дашборде           |
| LeadGen     | Отдельная вкладка: создание и распределение лидов; не ведёт заявки; не имеет доступа к чатам и контрактам              |

- Любой элемент интерфейса, недоступный роли, должен быть скрыт (не просто disabled), кроме случаев, когда требуется показать «заглушку».
- Dispatcher видит только свои чаты, лиды, заявки, контракты, клиентов и метрики (фильтр «только свои» не отключаемый).
- LeadGen работает в отдельной вкладке `Лидогенерация`.
- Переключение между ролями (только для Admin с доп. ролью тестирования) — **вне MVP**.

### 2.1 Детализация прав на действия

| Действие                                                      | Admin | Dispatcher   | LeadGen |
|---------------------------------------------------------------|:-----:|:------------:|:-------:|
| Управление пользователями / настройки                         | ✓     | ✗            | ✗       |
| Создание лида вручную / назначение                            | ✓     | ✓            | ✓       |
| Переписка, заявки, публикация, назначение                     | ✓     | ✓            | ✗       |
| Финансовые операции                                           | ✓     | Частично*    | ✗       |
| Изменение ставки комиссии                                     | ✓     | ✗            | ✗       |
| Контракты                                                     | ✓     | ✓            | ✗       |
| Дашборд / экспорт                                             | ✓     | Только своё  | ✗       |

\* «Частично» для Dispatcher: формирование/подтверждение инвойсов, отправка, отметка оплаты; редактирование ставок комиссии — запрещено.

## 3. Маршруты и структура приложения

Next.js App Router. Группы маршрутов через `(group)` используются для разных layout-ов.

```
app/
  (auth)/
    login/                      ─ без основного layout
  (app)/
    layout.tsx                  ─ sidebar + topbar + guards
    dashboard/
    tasks/
    calendar/
    inbox/                      ─ чат + 3-колоночный layout
      [dialogId]/
    funnel/                     ─ Kanban-воронка
    requests/
      [id]/
    contracts/
      [id]/
    clients/
      [id]/
    tutors/
      [id]/
    dispatchers/                ─ Admin
      [id]/
    leadgen/                    ─ вкладка LeadGen
    scripts/                    ─ Admin
    settings/                   ─ Admin
      templates/
      subjects/
      invoices/
      sla/
      auto-actions/
      channels/                 ─ привязка TG/WA/Viber/FB аккаунтов
      funnel-stages/
      rejection-reasons/
    profile/                    ─ все роли
  not-found.tsx
  error.tsx
  global-error.tsx
middleware.ts                   ─ auth-guard, редирект на /login
```

Главный layout группы `(app)` — левая навигация + верхняя полоса (поиск, уведомления, профиль). Внутри разделы имеют собственные sub-layouts (например, Inbox — 3-колоночный).

## 4. Общие UI-компоненты

Минимально необходимый набор переиспользуемых компонентов (должны быть зафиксированы в Storybook):

- **Button** (primary / secondary / ghost / destructive / icon).
- **Input / Textarea / Select / Combobox / DatePicker / TimePicker / DateRangePicker / NumberInput / PhoneInput / CurrencyInput / FileUpload**.
- **FormField** — обёртка с label/description/error.
- **Table** — сортировка, фильтры в заголовках, пагинация серверная, выбор строк, кастомные ячейки, «пустое состояние», сохранение состояния в URL.
- **Tabs, Breadcrumbs, Tooltip, Popover, Dialog (Modal), Drawer, Sheet, Dropdown, Menu, Toast, Badge, Tag, Avatar, Skeleton, EmptyState, Alert, Banner, Stepper, ProgressBar**.
- **RoleBadge**, **StatusBadge** (унифицирует цвет/текст по статусу).
- **MoneyLabel** — форматирование валюты и знака (грн).
- **PhoneLink**, **TelegramLink**, **WhatsAppLink** — быстрые действия.
- **ChannelIcon** — иконка канала коммуникации (TG / WA / Viber / FB / IG).
- **CommentEditor** — редактор скриптовых сообщений с вставкой переменных шаблона.
- **CountdownTimer** (SLA-таймер, таймер пробного).
- **ConfirmDialog** — подтверждение деструктивных действий.
- **AuditTrailItem** — блок истории (каждое действие с автором и временем).
- **CommandPalette** (Ctrl+K) — быстрый поиск по клиентам/заявкам/репетиторам (nice-to-have).

Принципы:
- Все состояния компонента должны иметь хотя бы один визуальный индикатор (loading, disabled, error, empty, partial).
- Длинные списки — виртуализация (react-virtual).
- Клавиатурная навигация — для форм, таблиц, модалок, Kanban.

## 5. Экраны

### 5.1 Логин
- Маршрут `/login` (группа `(auth)`, без основного layout).
- NextAuth credentials provider: email + пароль; кнопка «Войти».
- «Забыли пароль?» → отправка ссылки на email (если реализуется в MVP).
- После успешного входа — редирект на `/dashboard` (Admin/Dispatcher) или `/leadgen` (LeadGen).

### 5.2 Дашборд
- Верхняя полоса фильтров: период, предмет, диспетчер (последний — только Admin).
- Блоки метрик (см. раздел 12 бизнес-документа):
  - **Конверсия воронки** (funnel-chart: Чат → Лид → Заявка → Пробный → Успех).
  - **Скорость**: Лид → Публикация, Публикация → Назначение (гистограммы/числовые плитки).
  - **Финансы**: Выручка / Прибыль по предметам / диспетчерам / репетиторам (стек-бар, таблица).
  - **Операционные**: Лидов за период, просрочки SLA, счета.
  - **Репетиторы**: Конверсия пробных, Количество учеников.
- Кнопка **Экспорт** (CSV / XLSX) с учётом активных фильтров.
- Для Dispatcher — данные автоматически обрезаются до «только своих».

### 5.3 Задачи
- Таблица/список задач (автозадачи «Нужен фидбек», созданные вручную, задачи SLA-эскалации и т. д.).
- Столбцы: заголовок, клиент/заявка (ссылка), тип, срок, приоритет, статус, автор.
- Фильтры: мои / все (Admin), статус, тип, срок.
- Действия: открыть контекст (переход в заявку/чат), отметить выполненной, перенести срок, закрыть с причиной.

### 5.4 Календарь
- Представления: месяц / неделя / день / лист.
- События: пробные уроки, регулярные занятия по контрактам, напоминания по оплатам инвойсов.
- Цветовая маркировка: пробные, контрактные, разовые, задачи.
- Клик по событию — боковая панель с деталями и переходом к заявке/контракту.
- Создание события вручную — только связанное с существующей заявкой/контрактом.

### 5.5 Inbox (Чат) + Панель привлечения

Основной рабочий экран диспетчера. **Трёхколоночный layout**:

#### 5.5.1 Левая колонка — список диалогов
- Поиск по имени/телефону/ID.
- Фильтры: все / непрочитанные / требуют ответа / по каналу (TG / WA / Viber / FB / IG) / без лида.
- Карточка диалога: аватар, имя, канал-иконка, последнее сообщение, время, счётчик непрочитанных, бейдж SLA.
- Сортировка: по времени последнего сообщения.
- Индикатор «печатает» (если канал поддерживает).

#### 5.5.2 Центральная колонка — чат
- История сообщений с группировкой по дате.
- Поддержка медиа: текст, фото, видео, голосовые, документы, стикеры (read-only для тех, которые не поддерживаются на отправку).
- Индикаторы доставки/прочтения (в зависимости от канала).
- Поле ввода: текст, вложения, эмодзи, вставка скрипта из справочника.
- Кнопка «Шаблоны» — быстрый выбор из справочника шаблонов сообщений с подстановкой переменных.
- Действия с сообщением: копировать, переслать (если поддерживается каналом), процитировать.
- Переключатель канала, если с клиентом несколько каналов связи.

#### 5.5.3 Правая колонка — контекстная панель
Панель адаптируется к стадии диалога/заявки:

| Стадия              | Содержимое панели                                                                    |
|---------------------|--------------------------------------------------------------------------------------|
| Новый диалог        | Кнопка «Создать лид» + поиск/привязка к существующему клиенту                         |
| Лид создан          | Форма заявки (предмет, уровень, уроки/нед, график, цены, тип сделки, пробный)         |
| Заявка создана      | Превью поста + кнопка «Опубликовать»; подсвечен канал публикации                      |
| Опубликовано        | Список откликов с профилями репетиторов + кнопка «Назначить»                          |
| Репетитор найден    | Превью двух сообщений (клиенту + репетитору) + кнопка «Отправить»                     |
| Пробный назначен    | Таймер до пробного + кнопка «Результат пробного»                                      |
| Пробный проведён    | Блок «Успех» → закрытие по типу сделки (контракт/разовый) / «Неуспех» + форма фидбека |

- Постоянно видимые блоки: карточка клиента (ФИО, телефон, источник, кол-во активных контрактов), список активных заявок клиента, история действий.
- Свёртываемые секции.

#### 5.5.4 Инициация переписки
- Кнопка **«Написать первым»** (в шапке Inbox и из карточки клиента/лида): диалог с выбором канала и ввод ID/телефона.
- После успешной инициации — диалог появляется в Inbox диспетчера.

### 5.6 Kanban-воронка заявок
- Каждая заявка — карточка, движется по колонкам (этапам).
- Этапы настраиваются администратором (Название, Скрипты, Цвет, SLA).
- Стандартные этапы: `Формирование` → `Поиск репетитора` → `Репетитор найден` → `Пробный назначен` → `Пробный проведён` → `Закрыто (успех)` / `Закрыто (отказ)`.
- Тип сделки (контракт/разовый) — бейдж на карточке, определяет канал публикации.
- Карточка заявки: клиент, предмет, цена, тип сделки, ответственный, SLA-таймер (с цветовой индикацией просрочки).
- Drag-n-drop разрешён только по правилам перехода: вперёд — свободно, назад — запрещено (кроме НЕУСПЕХ → «Новый поиск»).
- Переход на «Отказ» — с обязательной причиной (модалка со справочником причин).
- Фильтры над доской: предмет, диспетчер (Admin), период, тип сделки.
- Для больших колонок — виртуализация + счётчик «показано N из M».

### 5.7 Контракты
- Список контрактов (таблица): клиент, репетитор, предмет, расписание, цены, статус (активен / на паузе / закрыт), долг клиента, долг репетитора.
- Фильтры: статус, клиент, репетитор, предмет.
- Действия: пауза / возобновление / закрытие (с сохранением активных неоплаченных инвойсов).
- Страница контракта:
  - Шапка: статус, участники, предмет, расписание, цены, ставки комиссии (ставки редактирует только Admin).
  - Блок «Еженедельный учёт»: ввод кол-ва проведённых уроков по неделям.
  - Блок «Инвойсы»: список инвойсов клиента + репетитора, статусы (черновик/отправлен/оплачен/просрочен), действия (редактировать сумму, пропустить, отправить, отметить оплаченным).
  - История изменений.

### 5.8 Клиенты
- Таблица: ФИО, телефон, источник, тип клиента (автоматический: Новый / Обычный / Контрактный / Смешанный), активные заявки, долг, последнее взаимодействие.
- Поиск (включая дубликаты — подсветка совпадений по телефону).
- Страница клиента:
  - Карточка: контакты, все каналы связи (TG/WA/Viber/FB/IG), источник, теги.
  - Вкладки: Лиды, Заявки, Контракты, Инвойсы, История общения, Файлы.

### 5.9 Заявки
- Таблица: ID, клиент, предмет, уровень, тип сделки, этап воронки, репетитор (если назначен), цена, ответственный диспетчер, дата создания, SLA.
- Фильтры: по всем столбцам, группировки.
- Переход на страницу заявки — полная воронка, вся история, превью поста и текущих сообщений.

### 5.10 Репетиторы
- Таблица: ФИО, Telegram ID, предметы, статус (зарегистрирован / не зарегистрирован / заблокирован), рейтинг, кол-во активных контрактов, конверсия пробных.
- Фильтры: предмет, статус.
- Страница репетитора:
  - Карточка: контакты, предметы, опыт, комментарии диспетчера.
  - История откликов, назначений, пробных, контрактов.
  - Финансы: текущий долг перед сервисом, история инвойсов.
- Действия: создать профиль вручную, подтвердить регистрацию из бота, заблокировать.

### 5.11 Диспетчеры (Admin)
- Таблица: ФИО, email, статус, подключённые аккаунты мессенджеров (по колонкам — ✓/✗), нагрузка (кол-во активных диалогов/заявок).
- Страница диспетчера: базовые данные, привязанные TG/WA/Viber/FB-аккаунты (раздел настроек каналов открывается отсюда), права, история активности, личные настройки уведомлений (только просмотр).

### 5.12 Вкладка LeadGen
- Отдельный экран только для роли LeadGen (и Admin).
- Форма/таблица создания лидов: ФИО, телефон, источник, комментарий, назначенный диспетчер.
- Проверка дублей по телефону — предупреждение с привязкой к существующему клиенту/лиду.
- Таблица созданных лидов с фильтрами: мои / все (Admin), период, источник, назначенный диспетчер, статус.

### 5.13 Скрипты (справочник)
- Категоризированный список шаблонов сообщений (например: «Первое касание», «Запрос деталей», «Контакт репетитору»).
- CRUD (только Admin).
- Переменные в шаблонах: `{{client.name}}`, `{{request.subject}}`, `{{tutor.phone}}` и т. д. (список поддерживаемых переменных — зафиксировать отдельно).
- Предпросмотр с подстановкой примера.

### 5.14 Настройки (Admin)

#### 5.14.1 Шаблоны
- Отдельные наборы: публикация заявки (с разделением контракт/разовый), инвойс клиенту, сообщение репетитору при назначении, системные уведомления.
- Редактор с подстановкой переменных, предпросмотр.

#### 5.14.2 Справочник предметов
- Таблица: название, TG-канал для контрактных, TG-канал для разовых, ставка комиссии клиента по умолчанию, ставка комиссии репетитора по умолчанию, активен / архив.
- CRUD; при удалении — проверка, что нет активных заявок.

#### 5.14.3 Инвойсы
- День недели формирования инвойсов.
- Срок ожидания оплаты (по умолчанию 2 дня).
- Шаблон номера инвойса.

#### 5.14.4 SLA
- Для каждого этапа воронки: таймер эскалации (мин/час), кому эскалировать (Admin / старший диспетчер).
- Общий SLA «ответ на новый диалог за N минут».

#### 5.14.5 Автодействия
Тумблеры:
- Автоотправка контактов при назначении репетитора.
- Автосмена статуса при указании даты пробного.
- Автоотправка инвойсов без предпросмотра.
- Автозадача «Нужен фидбек» после передачи контакта — **по умолчанию включена**.

#### 5.14.6 Каналы связи
- Привязка аккаунтов мессенджеров к диспетчерам (Telegram — Userbot + Bot компании, WhatsApp — Cloud API / Baileys / аналог, Viber — сторонний сервис, FB/IG — Graph API или сторонний интегратор).
- Статусы подключений, перевыпуск сессии, отзыв доступа.
- **UI этого раздела для Telegram будет детализирован отдельным модулем.**

#### 5.14.7 Этапы воронки
- CRUD этапов: Название, Цвет, SLA, Скрипты (какие подсказки показывать диспетчеру на этой стадии).
- Порядок этапов — drag-n-drop.
- Справочник причин отказа.

### 5.15 Профиль пользователя
- Имя, email, аватар.
- Смена пароля.
- Настройки уведомлений:
  - Категории в TG-бот (перечислить все события — см. раздел 7 бизнес-документа).
  - Тихие часы (период, когда push не приходят, кроме эскалаций).

## 6. Бизнес-процессы в UI

### 6.1 Создание лида
Источники:
1. Из входящего диалога (Inbox → «Создать лид»).
2. Вручную LeadGen/Dispatcher/Admin (форма на вкладке LeadGen или из карточки клиента).
3. Исходящий контакт: создание лида → из карточки лида кнопка «Написать в TG/WA/Viber» → инициация чата через Userbot.

Все пути проверяют дубликаты по телефону; при совпадении — предложение привязки к существующему клиенту.

### 6.2 Формирование заявки
Форма:
- Клиент (предзаполнен из лида).
- Предмет (выпадающий список справочника; подгружаются default-ставки комиссии).
- Уровень.
- Кол-во уроков в неделю.
- График занятий (чекбоксы дней + диапазоны времени).
- Цена в час (редактируется).
- Цена заявки (для разового типа).
- Тип сделки: **Контракт** / **Разовый** (радио-группа).
- Предпочтения по времени пробного.
- Комментарий.
- Кнопки: «Сохранить черновик» / «Сохранить и опубликовать».

Валидация на уровне формы + серверная. При выборе типа «Разовый» — поле «Цена заявки» обязательное.

### 6.3 Публикация
- Кнопка «Опубликовать» в заявке.
- CRM определяет канал по (предмет + тип сделки).
- Модалка предпросмотра: отрендеренный пост по шаблону + канал.
- Кнопки: «Редактировать пост» (разовое переопределение текста) / «Опубликовать».
- Статус заявки автоматически переходит в «Поиск репетитора».

### 6.4 Отклики репетиторов
- Список откликов в панели заявки (или отдельный виджет в карточке).
- Для каждого отклика: аватар, имя (или пометка «Не зарегистрирован»), Telegram-ник, метрики (конверсия пробных, кол-во учеников, рейтинг), текст отклика, время.
- Действия на отклике:
  - **Назначить** (для зарегистрированных) — переход к форме назначения.
  - **Зарегистрировать и назначить** (для незарегистрированных) — всплывает форма создания профиля репетитора с предзаполнением.
  - **Скрыть** (пометить как нерелевантный).

### 6.5 Назначение репетитора
Модалка:
- Превью сообщения клиенту (через диспетчера в чате).
- Превью сообщения репетитору (через диспетчера в чате) — контакты клиента, предмет, цена, инфо о пробном.
- Оба превью редактируемы.
- Кнопка «Отправить». Если автоотправка контактов включена — шаг превью можно пропустить автоматически (по настройке).
- После отправки:
  - Создаётся событие «Пробный урок» в календаре.
  - Создаётся автозадача «Нужен фидбек» диспетчеру (таймер — настраивается Admin).
  - Статус заявки → «Пробный назначен».

### 6.6 Фиксация результата пробного
Модалка:
- Радио: Успех / Неуспех.
- Для успеха:
  - Контракт — открывается форма создания контракта (п. 6.7).
  - Разовый — открывается блок контроля оплаты комиссии (п. 6.8).
- Для неуспеха:
  - Причина (справочник).
  - Действие: «Новый поиск репетитора» (возврат заявки в этап публикации с сохранением истории) / «Закрыть отказом».
- Отметить наличие фидбека от клиента (свободный текст).

### 6.7 Создание контракта
Форма (предзаполнена из заявки):
- Клиент, репетитор, предмет.
- Расписание занятий.
- Цена за урок.
- Ставка комиссии клиента, ставка комиссии репетитора (только Admin может менять значения, Dispatcher видит read-only, но может запросить изменение).
- Дата старта.
- Условия паузы/закрытия (информационно).
- Подтверждение → контракт в статусе «Активен», заявка — закрыта (успех).

### 6.8 Контроль оплаты (разовый заказ)
- Блок на странице заявки: «Ожидается оплата NNN грн от репетитора».
- Кнопка «Отправить напоминание» — открывает предпросмотр шаблонного сообщения репетитору с суммой (отправка через Userbot).
- Кнопка «Отметить оплаченным» → заявка закрыта (успех).
- Если прошло N дней — уведомление диспетчеру.

### 6.9 Еженедельный учёт и инвойсы (контракты)
- Виджет «Требует внимания» на странице контракта и в разделе «Контракты».
- Еженедельно, в настроенный день: уведомление диспетчеру «Внесите кол-во уроков за неделю».
- Форма ввода: кол-во уроков за прошлую неделю, комментарий.
- После ввода — генерация двух инвойсов:
  - Клиенту = N × ставка комиссии клиента.
  - Репетитору = N × ставка комиссии репетитора.
- Для каждого инвойса: превью, редактирование суммы (с логированием), отправка через Userbot.
- Статусы инвойса: Черновик → Отправлен → Оплачен / Просрочен / Пропущен.
- Кнопка «Пропустить» (болезнь/каникулы) — инвойс не формируется.
- Через N дней (по умолчанию 2) без оплаты — уведомление диспетчеру.
- Автоотправка без превью — по настройке Admin.

### 6.10 Замена репетитора
- Кнопка «Заменить репетитора» в активном контракте.
- Модалка: причина (справочник), выбор «Поставить контракт на паузу» (тумблер), комментарий.
- Создаётся новая заявка (предзаполнена данными контракта), отправляется в стандартную воронку.
- При успехе — контракт обновляется (репетитор меняется), старый репетитор получает уведомление через Userbot.

## 7. Уведомления

Два канала: **колокольчик в CRM** и **TG-бот диспетчера**. В UI — только первый; настройка TG-уведомлений — в профиле пользователя.

### 7.1 Колокольчик
- Счётчик непрочитанных в верхней полосе.
- Выпадающий список с группировкой по типу.
- Категории (совпадают с раделом 7 бизнес-документа):
  - Новый диалог / сообщение.
  - Новый отклик.
  - Нужен фидбек (после передачи контакта репетитору).
  - Просрочка SLA.
  - Инвойсы готовы к отправке.
  - Не внесено кол-во уроков за неделю.
  - Инвойс не оплачен (через 2 дня).
  - Эскалация: нет ответа на диалог за N минут (Admin/старший).
- Клик по уведомлению — переход в контекст.
- Возможность отметить всё прочитанным.

### 7.2 Toast-уведомления
- Входящие сообщения (если активная вкладка — без звука, при фоновой — со звуком; звук настраивается).
- Результаты действий (успех/ошибка).

### 7.3 Realtime
- Через WebSocket: новые сообщения, статус диалога, новые отклики, изменения воронки, события SLA.
- При потере соединения — баннер «Переподключение…», автоповтор с экспоненциальным backoff.

## 8. Финансовый модуль UI (краткое резюме)

- Разделение схем оплаты на уровне UI:
  - **Контракт**: клиент платит за уроки → репетитору (информативно, без инструментов); клиент платит комиссию → сервису (инвойс); репетитор платит комиссию → сервису (инвойс).
  - **Разовый**: репетитор платит фиксированную сумму (цена заявки) → сервису.
- Все ставки комиссии наследуются: Справочник предметов → Заявка → Контракт; на каждом уровне возможно переопределение (только Admin).
- UI должен явно показывать источник унаследованной ставки («Унаследовано из справочника предметов»).
- История изменений сумм инвойсов — всегда доступна (аудит).

## 9. Валидация, ошибки, пустые состояния

- Каждое пустое состояние списка/таблицы — визуал с описанием и CTA (если применимо).
- Все действия, меняющие состояние (удаление, закрытие контракта, отказ по заявке, отмена пробного) — через ConfirmDialog с чётким описанием последствий.
- Ошибки API — toast + в сложных случаях inline-описание под формой.
- Длительные операции (>300 мс) — loading-индикатор; >2 сек — skeleton.

## 10. Адаптив (минимум)

- ≥ 1440: полноценный 3-колоночный Inbox, Kanban с 5–7 колонками без скролла.
- 1280–1439: 3-колоночный Inbox с уменьшенной боковой панелью; Kanban с горизонтальным скроллом.
- < 1280: допускается ухудшенный, но рабочий режим (контекстная панель сворачивается в Drawer).

## 11. Производительность

- First Contentful Paint ≤ 1.5 сек на хорошем канале.
- Interactive ≤ 3 сек.
- При 200+ активных диалогах в Inbox — без лагов (виртуализация).
- При 500+ карточек в Kanban — без лагов (виртуализация по колонкам).
- Бандл основного чанка ≤ 350 КБ gzip (`next build` + `@next/bundle-analyzer`).
- Server Components для списочных экранов (клиенты, заявки, контракты) — минимизируют клиентский JS.

## 12. Доступность

- Все интерактивные элементы — фокусируемые, с видимым focus-ring.
- Клавиатурные шорткаты (минимум):
  - `Ctrl+K` — command palette.
  - `Ctrl+Enter` — отправка сообщения.
  - `N` — новый лид (из Inbox).
  - `G` + `I/F/C/K` — переходы.
- Соответствие WCAG AA для основных элементов.

## 15. Критерии приёмки (фронтенд)

- Все экраны раздела 5 реализованы с состояниями loading / empty / error / success.
